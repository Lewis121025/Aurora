# 01_Memory_Engine 原理与设计内参 (Design Review)

## 锚定源码：`aurora/core/memory/engine.py`
在这个训练营中，你的目标是用 Numpy 重构 Aurora 主引擎中的“遗忘算法机制”。

---

### 🤔 业务悖论：为什么我们需要“概率性遗忘”？

当一个用户的输入通过 `Embedding`（例如，被翻译成 1536 维度的 `[0.02, -0.44... 0.11]`）进入引擎后，引擎利用 `Cosine Similarity` 将它和系统过往数万条记忆进行了比对，得出一个 **VoI (Value of Information，信息价值分数)**。

假设分数范围在 `0` 到 `1` 之间：
* `0.9` -> 极大反常事件（比如用户突然说“我想辞职”）。
* `0.1` -> 毫无价值的废话（比如用户回了个“好的”）。

#### ❌ 反面教材 (Anti-Pattern：硬截断)
最直觉、最常见的平庸工程师的写法：
```python
voi_score = compute_voi()
if voi_score > 0.5:
    db.save()  # 大于 50 分无脑保存
else:
    db.drop()  # 小于 50 分直接永远丢弃
```
**致命缺陷：信息孤岛与死板剥夺**
在长期的人工语言心理学中，很多低分值的“微小寒暄（比如连续七天抱怨天气不好）”如果能被保存几条，会在未来拼凑出一幅极其完整的情绪图谱。如果用死板的硬截断阈值，这七天的微小情绪由于都低于 `0.5` 会被一刀切全部杀掉，导致系统产生巨大的“盲区断层”。这就是死板机器人的通病。

#### ✅ 架构方案（Thompson Sampling 概率化思想）
Aurora 的内核，甚至是当今世界上最好的一批广告推荐系统，使用的都是 **带偏见的抛硬币 (Thompson Sampling 等概率采样学的变体)**。

具体逻辑：
如果一句废话算出来的价值只有 `0.1`（很可怜）：
系统并不会判它死刑。系统会调用底层的一个随机掷骰子算子 `random()`（产生 0 到 1 之间的小数）。
如果掷出的骰子结果碰巧落在了 `0` 到 `0.1` 的小概率区间内，系统就会**放行**这句废话，把它当做正式记忆记录下来！

**核心架构意义：**
这就是人类常说的 **“机缘巧合的潜意识小记忆”**。即便价值极低（丢弃概率极高），我们依然留有一种算法级别的绝对宽容度去“偶发性收录”。这种极其微薄的偶发，构成了让用户觉得系统“像活人”的终极血肉感。
