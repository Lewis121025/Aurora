# ADR 001: 配置常量模块化重组

## 状态

已接受 - 2026-03-08

## 背景

原有的 `aurora/core/constants.py` 文件存在以下问题：

1. **单一文件过大**：595 行代码，50+ 个常量，难以维护
2. **缺少组织结构**：常量按注释分组，但没有模块级别的隔离
3. **职责不清**：检索、存储、一致性、演化等不同关注点混在一起
4. **缺少文档**：大部分常量缺少设计依据说明
5. **难以测试**：无法针对特定功能模块进行配置测试

## 决策

将 `constants.py` 拆分为按功能模块组织的配置包：

```
aurora/core/config/
  __init__.py          # 向后兼容的统一导出
  retrieval.py         # 检索相关常量
  storage.py           # 存储和容量相关常量
  coherence.py         # 一致性检查相关常量
  evolution.py         # 演化和反思相关常量
  identity.py          # 身份和关系相关常量
  knowledge.py         # 知识分类相关常量
  query_types.py       # 查询类型检测相关常量
  numeric.py           # 数值稳定性和基础常量
```

### 设计原则

1. **单一职责**：每个模块只包含一个功能域的常量
2. **向后兼容**：通过 `__init__.py` 重新导出所有常量，现有代码无需修改
3. **渐进迁移**：提供迁移脚本，支持逐步迁移到新结构
4. **文档优先**：每个常量都有注释说明用途和设计依据

### 迁移策略

1. **阶段 1**：创建新的模块化结构，保留旧的 `constants.py`
2. **阶段 2**：更新 `constants.py` 为重新导出层（向后兼容）
3. **阶段 3**：逐步迁移核心模块到新导入方式
4. **阶段 4**：废弃 `constants.py`，移除兼容层

## 后果

### 正面影响

- **可维护性提升**：每个模块 < 200 行，易于理解和修改
- **职责清晰**：配置按功能域隔离，修改影响范围明确
- **测试友好**：可以针对特定模块进行配置测试
- **文档改善**：每个模块有清晰的文档说明
- **扩展性好**：新增配置时有明确的归属位置

### 负面影响

- **导入路径变长**：从 `from aurora.core.constants import X` 变为 `from aurora.core.config.retrieval import X`
- **迁移成本**：需要更新现有代码的导入语句
- **学习曲线**：新开发者需要了解配置的模块划分

### 缓解措施

- 提供自动迁移脚本 `aurora/scripts/migrate_constants.py`
- 保持向后兼容层，给予充足的迁移时间
- 在文档中明确说明配置的模块划分原则
